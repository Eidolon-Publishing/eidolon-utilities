# Plan

Improve the tween timeline / sequencer / executor architecture to deliver **immediate cancellation**, **clear outcome semantics (cancel vs fail)**, **consistent error-policy behavior (including hooks)**, and **stronger validation**, while keeping the design small and backwards-compatible for existing callers and registered tween executors.

---

## Requirements

- Cancellation is **prompt** even when a step is currently awaiting in-flight tweens.
- Executors can **opt-in to cooperative cancellation** (AbortSignal) without breaking existing signatures.
- Timeline execution returns **structured outcomes** so callers can distinguish:
  - cancelled by user / replaced-by-name
  - runtime failure (exception, invalid state)
  - completed (optionally “completed with entry errors” under `errorPolicy="continue"`)
- `errorPolicy="continue"` applies to:
  - entry execution failures
  - hook failures (`beforeAll`, step `before`, step `after`, and completion callbacks)
- Sequence JSON validation supports:
  - **structural validation** (existing)
  - **semantic validation** (entry type exists + `validator(params)` passes) before running/broadcasting when possible
- Socket sync is robust against **race conditions** (e.g., “cancel old run cancels new run” when reusing the same timeline name).

---

## Scope

- In:
  - Timeline engine (`TweenTimeline`, `TimelineHandle`)
  - Sequence compile/validate pipeline (`schema.js`, `sequence.js`)
  - Executor contract via registry docs and normalization (`registry.js`)
  - Socket start/cancel payload semantics (minimal, additive)
- Out:
  - Rewriting all tween executors beyond adding cooperative cancellation hooks where straightforward
  - Major re-org of unrelated tween/lighting logic
  - Adding complex scheduling features (priority queues, dependency graphs, etc.)

---

## Current Behavior (baseline summary)

Key observed behaviors in the current code:
- Cancellation uses an `AbortController`, but **does not interrupt** `await Promise.all(stepPromises)`; cancellation is only noticed between segments / before hooks / after hooks.
- Failures and cancellations both collapse to `false` (`TimelineHandle.finished: Promise<boolean>`), making them indistinguishable.
- Hook exceptions currently **fail the timeline** even under `errorPolicy="continue"`.
- `validateSequenceJSON` is structural only; it does not validate registered types or per-type param validators prior to run.

---

## Recommended API Contracts (backward-compatible)

### 1) Executor `execute()` contract (minimal change)
Keep the current signature shape, but standardize fields passed in `opts`:

- Current: `execute(params, opts) => Promise<boolean>`
- Recommended (compatible): `execute(params, opts) => Promise<boolean | ExecOutcomeLike>`

Where:
- `opts.signal?: AbortSignal` (new)
- `opts.commit`, `opts.startEpochMS`, existing `durationMS`, `easing` remain
- Executors *may* ignore `signal` safely.

Outcome normalization:
- `true` → `{ status: "ok" }`
- `false` → `{ status: "error" }` (legacy)
- Throw / reject:
  - if caused by abort (`signal.aborted` / AbortError-like) → `{ status: "cancelled" }`
  - otherwise → `{ status: "error", error }`

This lets you incrementally upgrade executors without breaking old ones.

### 2) Timeline handle result surface
Keep `finished` for backward compatibility, add a richer result:

- `handle.finished: Promise<boolean>`
  - resolves `true` when the timeline reaches the end **without fatal failure/cancel**
  - resolves `false` on **cancel** or **fatal failure**
  - (Under `errorPolicy="continue"`, entry failures do *not* make it fatal; `finished` stays `true`.)
- New: `handle.outcome: Promise<TimelineOutcome>`
  - `TimelineOutcome = { status: "completed" | "cancelled" | "failed", hadErrors?: boolean, errors?: Array<...>, cancelledReason?: string }`
  - `hadErrors=true` means “completed but some entries/hooks failed under continue policy”.

### 3) Cancellation semantics
Standardize cancellation reasons:
- `"user"` (explicit cancel)
- `"replaced"` (same-name registry replacement)
- `"socket"` (remote cancel)
- `"parent"` (cancel propagated from parent timeline, e.g. sequence-in-sequence)

Expose on handle:
- `handle.cancel(reason = "user")`

### 4) Socket sync: introduce a per-run identifier (fixes a real race)
Add `runId` to both start and cancel envelopes:
- Start (`SOCKET_TYPE_SEQUENCE` payload): `{ name, runId, data, startEpochMS }`
- Stop (`SOCKET_TYPE_SEQUENCE_CANCEL` payload): `{ name, runId, reason?: "cancelled" | "failed" }`

Backward compatibility:
- If a cancel message arrives **without** `runId` (old sender), cancel the current run by `name` (legacy behavior).
- If `runId` exists, only cancel if it matches the currently running runId for that name.

This prevents “late cancel of old run kills new run”.

### Phase 5 — Propagate cancellation through nested `sequence`
Rationale: once `signal` exists, nested timelines should obey it.

- Socket stop semantics:
