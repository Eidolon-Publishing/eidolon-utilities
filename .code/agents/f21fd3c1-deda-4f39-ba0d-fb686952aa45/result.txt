# Plan

Improve the timeline/sequencer/executor flow to make cancellation deterministic, expose distinct outcomes, apply error policy consistently, and validate semantic tween params before execution, while keeping existing public behavior callable and minimizing breakage.

## Requirements
- Implement immediate, in-flight cancellation via shared abort signaling.
- Preserve backward compatibility for existing `Timeline` builder API and current return shape (`handle.finished` as boolean promise) while adding richer outcome inspection.
- Make `errorPolicy` deterministic and predictable across step hooks and entry execution.
- Validate sequence entries against registered tween validators before running, not only structural schema shape.
- Keep changes scoped and production-safe for Foundry multiplayer socket sync.

## Scope
- In: `scripts/tween/core/timeline/*`, `scripts/tween/core/registry.js`, `scripts/tween/core/executors/*`, `scripts/tween/hooks.js`, and targeted tests for timeline/executor paths.
- Out: other module systems outside tween (unless needed for shared test utilities/mocks).

## Phase 1 — Lock the contract first
1. Define a canonical execution outcome model and map it to legacy compatibility behavior (`completed` vs `failed` vs `cancelled`) without changing current public entry points.
2. Add explicit cancellation/error reason categories for timeline internals: e.g. `cancelled`, `entry_error`, `hook_error`, `validation_error`, and `runtime_error`.
3. Decide that hook callbacks are controlled by the same policy as entry execution (`errorPolicy`), with a clear continuation matrix to resolve the current asymmetry.

## Phase 2 — Add structured outcome + compatibility bridge
1. Introduce a richer result surface on `TimelineHandle` while retaining `handle.finished` as `Promise<boolean>`.
2. Keep `handle.finished` backward compatible (`true` for non-fatal completion, `false` for terminal failure/cancel), and add new read APIs such as `handle.result`/`handle.status`/`handle.reason` for precise classification.
3. Ensure `onComplete`/`onCancel` semantics remain callable as today, and consider adding optional callback arg with `{status, reason, errors}` for richer handling without breaking existing zero-arg usage.

## Phase 3 — Implement abort propagation for in-flight entries
1. Thread a shared `AbortSignal` from `TweenTimeline.run()` through segment execution to every entry execute call and to helper `cancellableDelay`.
2. Update all built-in executors to accept optional abort context and stop their internal animation promptly when aborted:
   - `scripts/tween/core/executors/light-color.js`
   - `scripts/tween/core/executors/light-state.js`
3. For `sequence` executor (`scripts/tween/core/executors/sequence.js`), pass the same cancellation signal into nested timeline runs so cancellation is transitive.
4. Add timeline-level abort cleanup that cancels pending delays, non-detached execution waits, and any per-step promises via signal rejection/abort short-circuit.

## Phase 4 — Fix error policy consistency (including hooks)
1. Apply the same catch-and-continue behavior to `beforeAll`, step `before`, and step `after` under `errorPolicy='continue'`.
2. Under `errorPolicy='abort'`, preserve fail-fast behavior on first hook/runtime failure and mark outcome as failure with context.
3. Under `continue`, do not abort the whole timeline for recoverable hook/entry failures; collect error details and continue where possible.
4. Keep boolean legacy outcome stable enough for existing callers while exposing the richer classification for new clients.

## Phase 5 — Strengthen schema/compile validation
