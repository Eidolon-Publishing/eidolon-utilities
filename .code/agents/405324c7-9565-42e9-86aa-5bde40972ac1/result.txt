# Plan

Improve the timeline/sequence/executor stack by making cancellation cooperative and immediate, making outcomes explicit (completed vs cancelled vs failed), aligning `errorPolicy` behavior across entries and hooks, and adding semantic validation for sequence entries before execution. The approach is incremental and backward-compatible so existing callers still work while new contracts are adopted.

## Requirements

- Cancellation must stop in-flight work as quickly as executors can cooperate.
- Timeline outcomes must distinguish `cancelled` from `failed`.
- `errorPolicy` must have clear, consistent behavior for hooks and entries.
- Sequence validation must include semantic param validation against registered tween validators before run.
- Existing external API consumers should not break immediately.

## Scope

- In: Timeline runtime contracts, sequence compilation/validation, executor interface evolution, API surface in tween hooks, tests, docs/migration notes.
- Out: New tween features, redesign of unrelated animation systems, major socket protocol rewrite.

## Ordered Phases

1. Phase 1: Define runtime contracts and compatibility layer.  
Rationale: Locking the contract first avoids piecemeal behavior drift.
- Define a `TimelineOutcome` model with statuses: `completed`, `cancelled`, `failed`.
- Define failure shape: `{ status: "failed", error, phase, stepIndex?, entryType? }`.
- Define cancellation shape: `{ status: "cancelled", reason? }`.
- Keep `TimelineHandle.finished` behavior (`Promise<boolean>`) for compatibility.
- Add new `TimelineHandle.result` (`Promise<TimelineOutcome>`) as the canonical API.
- Document deprecation path for boolean-only consumers.

2. Phase 2: Propagate cancellation into executors (cooperative abort).  
Rationale: Immediate cancellation requires signal propagation, not only top-level checks.
- Extend executor options contract to include optional `signal: AbortSignal`.
- Update timeline execution to pass `signal` to each entry execute call.
- Update built-in executors to observe `signal` and terminate local animation early when aborted.
- Ensure delays and step-level waits short-circuit on abort without waiting for unrelated promises.
- Preserve behavior for third-party executors that ignore `signal` (best-effort cancellation).

3. Phase 3: Unify error semantics for hooks and entries.  
Rationale: `errorPolicy="continue"` should not unpredictably abort due to hook throws.
- Apply `errorPolicy` consistently to `beforeAll`, `before`, `after`, and entry execution.
- Define concrete behavior:
  - `abort`: first thrown/rejected hook or entry fails timeline (`failed`).
  - `continue`: record error and continue to next safe unit.
- For `before` hook failure under `continue`, skip that step’s entries and move to next segment.
- For `after` hook failure under `continue`, record and continue.
- For `beforeAll` failure under `continue`, record and continue timeline.
- Add centralized error recorder utility inside timeline engine to avoid repeated ad hoc catch logic.

4. Phase 4: Add semantic sequence validation pass.  
Rationale: Structural validity is insufficient; bad params should fail early and clearly.
- Keep `validateSequenceJSON` as structural validator.
- Add `validateSequenceSemantics` that checks:
  - tween type exists in registry (or explicit mode for deferred registration).
  - `tweenDef.validate(entry.params)` succeeds.
- Run semantic validation in sequence executor before compile/run.
- Run semantic validation in socket sequence handler before local execution.
- Return explicit validation failures mapped to `failed` outcomes (not `cancelled`).

5. Phase 5: Harden timeline lifecycle and socket outcome behavior.  
Rationale: Named timelines and cross-client sync need deterministic lifecycle rules.
- Ensure named timeline replacement cancellation emits explicit cancellation reason (e.g., `replaced-by-name`).
- Keep socket cancel message flow unchanged initially, but ensure local result is `cancelled`, not generic false.
- Avoid double-notification race when cancellation and failure overlap; first terminal outcome wins.
- Ensure `onComplete` runs only on `completed`; `onCancel` runs only on `cancelled`; add optional `onError` for `failed` (non-breaking addition).

6. Phase 6: Add tests (unit + integration-style mocks) and validate manually in Foundry.  
Rationale: These are behavioral changes with regression risk; tests must pin semantics.
- Unit tests for timeline core:
  - `finished` boolean compatibility and `result` status mapping.
  - cancel during long-running step returns `cancelled` promptly.
  - `abort` policy fails on hook throw and entry reject.
  - `continue` policy records hook/entry errors and proceeds per rules.
- Unit tests for schema/sequence:
  - structural-only validation still works.
  - semantic validation catches unknown type and invalid params.
- Integration-style tests with mocked registry/socket:
  - named timeline replacement cancels prior handle.
  - sequence socket handler validates semantics before run.
- Manual Foundry validation:
  - GM + one player client synchronized sequence start.
  - mid-step cancel from GM propagates and stops visibly.
  - invalid sequence payload logs clear validation error and does not partially run.
  - hook throw behavior matches `abort` vs `continue`.

7. Phase 7: Rollout and migration.  
Rationale: Keep production stable while guiding consumers to richer API.

## Risks and Mitigations

- Risk: Third-party executors ignore `signal`, causing non-immediate cancellation.  
Mitigation: Contract marks `signal` optional but recommended; built-ins comply first; timeline still aborts future segments immediately.
- Risk: Behavior change under `continue` for hook failures could surprise existing users.  
Mitigation: Document exact semantics and include migration note with examples.
- Risk: Added outcome complexity increases maintenance burden.  
Mitigation: Keep boolean compatibility and centralize status transitions in one place.
- Risk: Socket timing race between cancel and failure.  
Mitigation: first terminal outcome wins, with idempotent handle resolution.
