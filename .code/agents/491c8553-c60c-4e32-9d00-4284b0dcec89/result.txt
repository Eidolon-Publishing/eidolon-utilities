```js
  id: string,        // stable internal id (UUID/randomID)
  key: string,       // stable matching key used in rules/state (slug)
  label: string,     // display label
  values: string[],  // allowed values (unique, trimmed)
  default: string,   // must be in values
  order: number      // for UI ordering (optional but useful)
}
```

Why both `id` and `key`?
- `map-ui-presets` rules/state match on `key` strings; those must be stable and human-meaningful.
- Existing `eidolon` “variants” + light presets currently use **category IDs** as keys in stored selections (`computePresetKey` expects `{ categoryId: value }` in `scripts/light-presets/core/storage.js:297`). An `id` field lets you bridge old content without forcing immediate rewrites.

### 2) Compatibility read layer (map-ui-presets + legacy variants)
Add a small “source-of-truth resolver” layer (conceptual, no code here) used by both UI and runtime:
- `getEffectiveCriteria(scene)`:
  1) if `flags.eidolon-utilities.criteria` exists → use it
  2) else if `flags.map-ui-presets.criteria` exists → use it (read-only import mode)
  3) else if `flags.eidolon-utilities.objectVariants` exists → treat as legacy variants categories and offer migration
- `getEffectiveState(scene)`:
  1) prefer `flags.eidolon-utilities.state`
  2) else fallback to `flags.map-ui-presets.state` (read-only import mode)
  3) else default state derived from criteria defaults (do not guess values beyond defaults)

This provides “works immediately” compatibility even before running migrations, and keeps migrations user-controlled.

### 3) Placeable preset parity strategy (behavioral parity, not immediate editor parity)
Even if eidolon keeps its current light preset storage for a while, enforce parity at the *resolution* layer:
- Runtime resolution should accept “base + rules” with `criteria` objects and apply `map-ui-presets` merge ordering semantics.
- For eidolon light presets (currently “default + presets keyed by category selections”), plan a staged conversion to “base + rules”:
  - `default` → `base`
  - each stored preset entry `{categories, config}` → rule `{criteria, delta}` where `criteria` is translated into `{ criterionKey: value }` and `delta` is config diff vs base (similar to `convertLegacyPresets` approach in `/mnt/talanor.home/Foundry/prod/Data/modules/map-ui-presets/src/indexer.js:33`).

## Concrete Migration / Renaming Strategy (“variants” → “criteria”)

### A) Legacy eidolon `objectVariants` → eidolon `criteria`
Source: `flags.eidolon-utilities.objectVariants` list (`scripts/object-variants/core/storage.js:4`).

Proposed mapping:
- For each legacy category `{id, name, values[]}`:
  - `criterion.id = category.id` (preserves linkability to existing light presets keyed by categoryId)
  - `criterion.label = category.name` (fallback “Unnamed”)
  - `criterion.key = slugify(category.name)` **if unique**, else `criterion-${index}`; store the legacy id permanently in `id`
  - `criterion.values = uniqueTrim(values)`; if empty, seed with `["Standard"]` to avoid unusable criteria
  - `criterion.default = "Standard"` if present else first value
- Store a one-time migration marker:
  - `flags.eidolon-utilities.criteriaVersion = 1`
  - `flags.eidolon-utilities.migrations.objectVariantsToCriteria = { at: timestamp, fromCount, toCount }` (optional audit trail)

Critical compatibility bridge (for existing light presets):
- Continue accepting categoryId-keyed selections, but translate them through:
  - if selection key matches a criterion `id`, treat it as that criterion
  - else if selection key matches criterion `key`, treat it as that criterion
This avoids breaking existing `computePresetKey` behavior and enables a gradual shift toward `key`.

### B) map-ui-presets → eidolon
If a scene has:
- `flags.map-ui-presets.criteria`: import 1:1 into `flags.eidolon-utilities.criteria`, adding `id` fields.
- `flags.map-ui-presets.state`: import into `flags.eidolon-utilities.state`, but validate values against imported criteria; if invalid, clamp to default and report in migration summary.

No destructive writes to `map-ui-presets` flags during import (migration safety).

### C) Renaming “variants” naming in UI/strings
- Replace visible labels “Variants” with “Criteria” in Scene Config and any callouts (e.g., update light preset warnings that point to “Scene → Variants” in `scripts/light-presets/ui/AmbientLightConfigControls.js:171`).
- Keep internal backward-compat readers for a full release cycle after migration ships.
