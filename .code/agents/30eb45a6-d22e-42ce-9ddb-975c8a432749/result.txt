# Plan

Redesign the Scene Config Criteria tab as a compact authoring surface, while preserving `map-ui-presets` semantics and ensuring safe migration from both `eidolon-utilities` legacy `objectVariants` data and `map-ui-presets` criteria scenes. Keep day-to-day state manipulation in the floating window; this tab should be optimized for defining dimensions, values, defaults, and migration status.

## Requirements
- Preserve exact criteria/state behavior parity with `map-ui-presets` matching semantics from `src/resolver.js` and related state handling.
- Preserve placeable preset model parity: `base` + ordered `rules`.
- Keep Scene Config tab compact and authoring-focused.
- Preserve compatibility with existing `map-ui-presets` scenes and provide a safe migration path.
- Avoid overengineering and keep architecture maintainable.

## Grounded Baseline
- Current Scene tab is category-list CRUD only in `scripts/object-variants/ui/SceneConfigTab.js` and `templates/object-variants-scene-tab.html`.
- Current object variant storage is array-based in `scripts/object-variants/core/storage.js` (`flags.eidolon-utilities.objectVariants`).
- Current light preset authoring depends on those categories in `scripts/light-presets/ui/AmbientLightConfigControls.js`.
- `map-ui-presets` canonical criteria/state model is in `src/state.js`.
- `map-ui-presets` rule layering semantics are in `src/resolver.js`.
- `map-ui-presets` placeable update model (`base` + `rules`) is in `src/placeables.js`.

## Data Model Recommendation (Concrete)
- Canonical scene flags in `eidolon-utilities`:
- `flags.eidolon-utilities.sceneCriteriaVersion`: number, start at `1`.
- `flags.eidolon-utilities.sceneCriteria`: ordered array of `{ key, label, values, default }`.
- `flags.eidolon-utilities.sceneCriteriaState`: object map `{ [key]: value }`.
- Canonical placeable flags in `eidolon-utilities`:
- `flags.eidolon-utilities.presets`: `{ base: object, rules: [{ criteria: object, delta: object }] }`.
- Compatibility read order for scene criteria:
- Read `sceneCriteria` first.
- If missing, adapt `flags.eidolon-utilities.objectVariants` to criteria.
- If still missing, adapt `flags.map-ui-presets.criteria` and `flags.map-ui-presets.state`.
- Key policy:
- `key` is machine-stable and unique; editable only at creation (or explicit migration flow), not casual inline rename.
- `label` is user-facing and freely editable.
- Value policy:
- Trimmed, deduped, non-empty strings.
- `default` must always be in `values`.
- State policy:
- Ensure every criterion has a state value; repair missing values to criterion `default`.

## UI Interaction Model (Single Compact Tab)
- Top summary row:
- `Criteria: N`, `Values: M`, `Incomplete: K`, plus actions `Add Criterion`, `Import`, `Repair`.
- Criteria list:
- Dense row per criterion with `label`, `key badge`, `default chip`, `value count`, `actions`.
- Row expansion:
- Inline editor with `Label`, `Key` (readonly after create), tokenized `Values`, `Default` dropdown.
- Quick actions:
- `Duplicate`, `Delete`, `Move Up`, `Move Down`.
- Compact behavior:
- Rows collapsed by default.
- Auto-save on explicit row save action, not per keystroke.
- Validation shown inline in-row, no global blocking modal unless destructive.
- Migration affordance:
- If data is coming from legacy source, show a non-intrusive banner: `Imported from legacy format; review keys/defaults`.
- Scope boundary:
- No embedded runtime state switcher controls here; keep that in floating window.
